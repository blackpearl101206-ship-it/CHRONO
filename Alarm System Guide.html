01

Feature Model — Reminder vs. Alarm

Chrono ships two distinct notification primitives, and treating them as separate systems is the single most important architecture decision. A reminder is a soft nudge — one push, one banner, dismissible. An alarm is a hard interrupt — full-screen, looping audio, must be actively silenced. They live in different tables, use different OS APIs, and have different reliability guarantees.

Soft nudge

Message Reminders

Fire before a task starts. Non-blocking. Delivered as OS notifications + in-app toasts. Users can snooze the reminder — the underlying alarm is untouched.

  • Fires at startTime − offset (5, 10, 15, 30, 60 min)
  • OS-level local notification (no server round trip)
  • Auto-dismisses after 8 seconds if untouched
  • Tap → deep-link into task detail
  • One-shot: does not repeat unless task does
Hard interrupt

Custom Alarm System

Fires at the exact start time. Full-screen wake-up UI with looping audio + vibration. User must Stop or Snooze. Survives app termination and device reboot.

  • Triggers at startTime down to the second
  • Exact-time OS alarm (bypasses Doze / battery saver)
  • Loops custom audio + vibration pattern
  • Snooze × N or Stop; auto-stop after 5 min
  • Reschedules itself for tomorrow if repeat is set
The one rule that keeps you sane: reminders are notifications, alarms are scheduled OS events. Never try to build an alarm on top of a plain push. Push delivery is best-effort — an alarm at 06:00 needs to be a promise, not a hope.

Firestore variant (Flutter + Firebase)

If you're on Firestore, keep the same shape but as three collections. Put alarm_preferences as a subcollection of schedules — you get automatic parent scoping and simpler security rules.

// Firestore collection layout
users / {uid} /
  schedules / {scheduleId}          // document
    alarm_prefs / {prefId}          // subcollection, 1 doc
  audio_files / {audioId}           // user's uploaded tones

// Security rule sketch
match /users/{uid}/{document=**} {
  allow read, write: if request.auth.uid == uid;
}

Why the next_fire_at column? A cheap partial index (where alarm_enabled = true) makes the "what alarms fire in the next 60 seconds?" query O(log n) — critical for the server-side cron that hydrates OS-level schedulers when the app comes back online.
04

System Architecture & Background Logic

The hard part isn't the UI — it's making the alarm ring when the app is gone. Here's the mental model: the app is a producer, the OS is the executor. Chrono never runs the alarm itself. It hands a signed contract to the OS ("wake me at 09:00:00, play audio X, run this Activity"), and the OS honors it even if the app has been swiped away, garbage-collected, or the device rebooted.

The three lifecycle states, and what happens

  • App in foreground — Chrono creates the alarm, writes it to the local DB, and hands it to the OS scheduler. The scheduler's callback runs an in-process handler; UI is already loaded, so we just push a route to the full-screen alarm scene.
  • App backgrounded or killed — The OS still holds the exact-time alarm. On Android it wakes a BroadcastReceiver which starts a foreground service that plays audio + shows a full-screen intent. On iOS a UNNotification fires with a critical-alert sound; tapping it cold-launches Chrono into the alarm scene.
  • Device rebooted — Android drops all pending alarms on reboot. A BOOT_COMPLETED receiver reads the local DB and re-arms every future alarm_preferences.next_fire_at. iOS persists notifications through reboot — no action needed.

Why not "just use push notifications"?

Push (FCM/APNs) is best effort. Google can delay a push by 30 minutes to save battery. Apple can coalesce them. For a reminder, that's fine — a 30-second miss on a soft nudge is invisible. For an alarm, it's a broken product. Alarms must use OS-level exact-time schedulers (AlarmManager on Android, UNUserNotificationCenter on iOS) which are guaranteed to fire on time regardless of network state.

Android's 500-alarm cliff. The SCHEDULE_EXACT_ALARM permission caps at ~500 pending exact alarms per app. Never schedule "next 90 days at once" — only schedule the next fire, and let each alarm re-schedule its successor at fire time. Same for iOS's 64 pending notifications limit: rotate them.

REST API Contract

Nine endpoints cover every operation. Auth is Authorization: Bearer <jwt>. All responses are JSON. Errors follow { error: { code, message } }. Audio upload is a multipart/form-data POST — everything else is JSON.

MethodEndpointPurpose
GET/schedulesList today's schedules for the authed user (query: ?date=YYYY-MM-DD).
POST/schedulesCreate a schedule + default alarm_preferences row in one txn. Returns both.
PUT/schedules/:idUpdate title / start / end / notes / category / is_completed.
DEL/schedules/:idCascade-deletes the paired alarm_preferences. Cancels OS-level alarm client-side.
GET/schedules/:id/alarmRead the alarm preferences for a schedule.
PUT/schedules/:id/alarmUpdate reminder toggle, offset, alarm toggle, repeat_days, snooze, vibration, volume, audio_file_id.
GET/audio-filesList the user's uploaded audio + preset tones. Returns signed URLs valid for 24h.
POST/audio-filesUpload a new file. multipart/form-data; server validates MIME + size, stores blob, returns row.
DEL/audio-files/:idDeletes the blob + row. Any alarm referencing it falls back to the default preset.

Alarm Trigger Logic — Both Stacks

The exact plumbing that turns "user tapped save" into "phone screams at 06:00". Same pattern in both languages: compute the next fire time → hand it to the OS scheduler with a payload → OS invokes a receiver → receiver launches the alarm screen and starts audio.

Flutter — schedule + fire

// alarm_service.dart · using flutter_local_notifications + android_alarm_manager_plus
import 'package:flutter_local_notifications/flutter_local_notifications.dart';
import 'package:android_alarm_manager_plus/android_alarm_manager_plus.dart';

class AlarmService {
  static final _plugin = FlutterLocalNotificationsPlugin();

  static Future<void> scheduleAlarm({
    required String id,
    required DateTime fireAt,
    required String title,
    required String audioFilePath,
    required String vibrationPattern,
  }) async {
    final androidDetails = AndroidNotificationDetails(
      'chrono_alarm_channel', 'Alarms',
      importance: Importance.max,
      priority:   Priority.max,
      fullScreenIntent: true,      // wakes screen
      category: AndroidNotificationCategory.alarm,
      sound: UriAndroidNotificationSound(audioFilePath),
      playSound: true,
      audioAttributesUsage: AudioAttributesUsage.alarm,
      vibrationPattern: _vibPattern(vibrationPattern),
      ongoing: true,                 // user must dismiss
    );

    await _plugin.zonedSchedule(
      id.hashCode,
      title,
      'Tap to open',
      tz.TZDateTime.from(fireAt, tz.local),
      NotificationDetails(android: androidDetails),
      androidScheduleMode: AndroidScheduleMode.exactAllowWhileIdle,
      uiLocalNotificationDateInterpretation:
        UILocalNotificationDateInterpretation.absoluteTime,
    );
  }

  static Future<void> cancelAlarm(String id) =>
    _plugin.cancel(id.hashCode);
}

Flutter — boot receiver

// After device reboot, re-arm all future alarms from local DB
Future<void> onBootCompleted() async {
  final alarms = await db.alarmsWithFutureFireTime();
  for (final a in alarms) {
    await AlarmService.scheduleAlarm(
      id: a.id, fireAt: a.nextFireAt,
      title: a.title, audioFilePath: a.audioPath,
      vibrationPattern: a.vibration,
    );
  }

React Native — schedule + fire

// alarmService.ts · using @notifee/react-native
import notifee, {
  AndroidCategory, AndroidImportance,
  TriggerType, TimestampTrigger,
} from '@notifee/react-native';

export async function scheduleAlarm(a: AlarmPref & { fireAt: Date }) {
  const channelId = await notifee.createChannel({
    id: 'chrono-alarms',
    name: 'Alarms',
    importance: AndroidImportance.HIGH,
    sound: a.audioFileName,           // bundled in res/raw or app-private
    vibration: true,
    vibrationPattern: vibPattern(a.vibrationPattern),
  });

  const trigger: TimestampTrigger = {
    type: TriggerType.TIMESTAMP,
    timestamp: a.fireAt.getTime(),
    alarmManager: { allowWhileIdle: true },   // exact alarm
  };

  await notifee.createTriggerNotification(
    {
      id: a.id,
      title: a.title,
      body: 'Tap to open',
      android: {
        channelId,
        category: AndroidCategory.ALARM,
        importance: AndroidImportance.HIGH,
        fullScreenAction: { id: 'default', launchActivity: 'default' },
        pressAction:      { id: 'default', launchActivity: 'default' },
        loopSound: true,
        ongoing: true,
      },
      ios: {
        sound: a.audioFileName,
        critical: true,
        criticalVolume: a.volume / 100,
        interruptionLevel: 'timeSensitive',
      },
    },
    trigger,
  );
}

export async function cancelAlarm(id: string) {
  await notifee.cancelTriggerNotification(id);
}

React Native — full-screen alarm scene

// AlarmScreen.tsx · shown when the notification's full-screen intent fires
import TrackPlayer, { RepeatMode } from 'react-native-track-player';

export function AlarmScreen({ alarm }: { alarm: AlarmPref }) {
  useEffect(() => {
    (async () => {
      await TrackPlayer.setupPlayer();
      await TrackPlayer.add({
        url: alarm.audioPath, title: alarm.title, artist: 'Chrono',
      });
      await TrackPlayer.setRepeatMode(RepeatMode.Track);
      await TrackPlayer.setVolume(alarm.volume / 100);
      await TrackPlayer.play();
    })();
    return () => TrackPlayer.reset();   // on unmount
  }, []);

  return (
    <FullScreenAlarm
      time={alarm.fireAt}
      title={alarm.title}
      onSnooze={() => rescheduleIn(alarm.snoozeDurationMin)}
      onStop={() => { TrackPlayer.reset(); markStopped(alarm.id); }}
    />
  );
}

Audio upload — validate & store (Node/Fastify)

// server/routes/audio.ts
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
import ffprobe from 'ffprobe-static';
import { execFile } from 'node:child_process';

const MAX_BYTES = 10 * 1024 * 1024;    // 10 MB
const ALLOWED   = ['audio/mpeg', 'audio/wav', 'audio/aac', 'audio/mp4'];

app.post('/audio-files', { preHandler: authGuard }, async (req, reply) => {
  const file = await req.file();
  if (!ALLOWED.includes(file.mimetype))
    return reply.code(415).send({ error: 'unsupported_media_type' });

  const buf = await file.toBuffer();
  if (buf.length > MAX_BYTES)
    return reply.code(413).send({ error: 'file_too_large' });

  // probe duration with ffmpeg's ffprobe
  const duration = await probeDurationSec(buf);

  const key = `u/${req.user.id}/${nanoid()}.mp3`;
  await s3.send(new PutObjectCommand({
    Bucket: BUCKET, Key: key,
    Body:   buf,
    ContentType: file.mimetype,
  }));

  const row = await db.audioFiles.create({
    data: {
      userId: req.user.id,
      displayName: (req.body?.display_name ?? file.filename).slice(0, 120),
      storageUrl: signUrl(key),
      mimeType: file.mimetype,
      sizeBytes: buf.length,
      durationSec: duration,
    },
  });

  reply.code(201).send(row);
});
Snooze is just a re-schedule. When the user hits Snooze, cancel the current OS alarm, compute now + snooze_duration_min, and hand a fresh alarm to the scheduler with the same payload. Same code path as first-time scheduling.

Edge Cases & QA Checklist

Alarm apps live and die by these edge cases. Every item below has burned somebody. Ship with all of them tested on real hardware — emulators lie about background lifecycles.

Alarm fires when app is swiped away from recents.
Alarm fires after a device reboot (Android: BOOT_COMPLETED).
Alarm fires in Doze mode / battery saver (Android 6+).
Alarm fires with device on silent / DND (iOS critical alerts).
Snooze correctly cancels + reschedules; no duplicates.
Repeat days: alarm re-arms itself for next matching weekday.
Deleting the schedule cancels its OS alarm before deleting the row.
Editing start_time cancels the old alarm and schedules the new one.
Deleting an audio file falls back to preset for referencing alarms.
Upload rejected: MIME wrong · size > 10 MB · corrupt file.
Offline: uploaded audio still plays from app-private cache.
Timezone change (traveling): alarms fire at the intended local time.
Daylight saving: 02:30 on the "spring forward" day does not fire twice.
Permission denied: UI shows a clear "Grant permission" banner.
Alarm auto-stops after 5 minutes of no interaction; logs a "missed" event.
Screen locked: full-screen alarm intent bypasses the lock screen (Android).
